## Pipeline fusion

### Fully fused: filter â†’ map â†’ take

```ts
pipe(data, A.filter(x => x > threshold), A.map(x => x * 2), A.take(10))
```

All three operations fuse into a single loop. `take(10)` halts after 10 results. The loop never visits the rest of the array.

<FusionSpotlight />

<TabbedScalingChart category="filterâ†’mapâ†’take(10)" />

stopcock holds ~4M ops/s from n=100 to n=100K (the flat line). Every other library degrades linearly because they make three separate passes and allocate intermediate arrays. At 100K elements, native JS does 703 ops/s. Fusion does 4.2M.

### Partial fusion: filter â†’ map â†’ sort â†’ take

```ts
pipe(data, A.filter(x => x > 0), A.map(x => x * 2), A.sortBy((a, b) => a - b), A.take(5))
```

`sort` can't fuse because it needs the full array. `filter â†’ map` fuses into one loop, materializes the result, then `sort` runs, then `take` slices.

<TabbedBarChart category="filterâ†’mapâ†’sortâ†’take(5)" arraySize={10000} />

<TabbedScalingChart category="filterâ†’mapâ†’sortâ†’take(5)" />

The advantage shrinks because `sort` dominates the runtime. But partial fusion still helps. The fused `filter â†’ map` segment avoids an intermediate array and runs in one pass before handing off to `sort`.
